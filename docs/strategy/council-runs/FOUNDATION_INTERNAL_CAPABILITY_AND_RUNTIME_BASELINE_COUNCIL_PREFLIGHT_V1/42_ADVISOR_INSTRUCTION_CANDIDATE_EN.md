## Purpose and exact decision contract

Establish the current evidence-grounded state of Foundation as the shared canonical AI,
knowledge, evidence, judgment, and safety core for SIASIU, Cosmile, and future services.

The reviewed result must enable Leo to decide:

> Which current Foundation capabilities are trustworthy and reusable, which exact gaps
> deserve the next bounded implementation priority, and which capabilities should be
> retained, repaired, boundedly transferred, replaced, retired, deferred, or remain
> unverified so Foundation can provide an evidence-grounded AI/knowledge/judgment core to
> SIASIU and optional Cosmile AI surfaces without coupling ordinary commerce?

Evidence priority is tied to this approved near-term outcome envelope:

- SIASIU evidence-grounded consultation and judgment consumption;
- optional Cosmile AI consultation and Foundation product-snapshot boundaries, while
  ordinary commerce remains Foundation-independent;
- Foundation canonical product, brand, ingredient, claim, warning, safety, and provenance
  service capability;
- atomic product and governed-learning readiness as future structural/control questions
  only, not current build or activation authority.

P1 must verify whether these consumers and paths are current, partial, historical, dead,
absent, or unverified. Target fit or source presence alone must not create implementation
priority.

## Authority

Authorized after Leo approval:

- repository metadata, Git history, branch/HEAD/upstream, and relevant ancestry checks;
- read-only inspection of Foundation and exact historically relevant foundation-control
  source;
- read-only canonical Vault/knowledge inspection, including provenance and version pins;
- targeted read-only SIASIU and Cosmile adapter/consumption inspection after P1 admission;
- isolated, mission-owned, non-production build, test, API, retrieval, index, and
  synthetic-question execution after the applicable probe gate passes;
- exact evidence capture, Independent Review, Advisor final audit, documentation commits,
  and push to a docs-only branch.

Not authorized:

- product-code, canonical Vault, tracked configuration, lockfile, schema, or migration
  changes;
- shared, staging, production, or customer data/resources;
- real or re-identifiable PII;
- secret creation, disclosure, copying into evidence, or unapproved credential use;
- unapproved provider/network calls, public listeners, live flags, or production behavior;
- Memory V3, sender/intake, scheduler/watcher, autonomous job, persistent candidate store,
  promotion endpoint, serving-index mutation, or canonical write activation;
- Foundation becoming a synchronous dependency of ordinary Cosmile commerce;
- architecture or ownership transfer, risk acceptance, implementation, merge, deployment,
  or automatic next mission.

## P2 — Selected Foundation-local verification

Run only P1-selected, safely admitted slices. Possible evidence classes include build,
tests, local API, retrieval/index, reasoning/judgment/safety, model/provider where separately
authorized, memory-boundary, and synthetic question traces.

Every probe needs a tier-appropriate record covering exact pin/command, intended claim,
principal, allowed reads/writes, temporary paths, dependencies, listeners/ports, network,
provider/account/environment, credential reference without value, synthetic inputs,
resource/request/token/time/cost caps, evidence capture, abort owner, teardown, and

Static read-only checks use a minimal record. Local build/test/index probes require
filesystem/resource/cleanup fields. Network/provider/mutation-capable probes use the full
deny-by-default record.

Missing prerequisites before execution yield `NOT_RUN`, `UNVERIFIED`, or
`PROVIDER_EXECUTION_NOT_RUN`. An observed boundary breach, unexpected external/canonical
write, secret or PII exposure, shared/production contact, public listener, Memory/promotion
activation, resource/cost breach, or failed cleanup immediately stops the affected phase
and returns `HOLD` with evidence.

## P4 — Evidence freeze, Independent Review, options, and closure

1. Freeze a content-addressed current-state evidence snapshot.
2. Independent Reviewer inspects actual relevant code, commands, tests, runtime probes,
   traces, failures, unknown routes, boundary compliance, and cleanup. Reviewer does not
   approve architecture, ownership, Legal status, security risk, provider choice, or
   implementation.
3. Responsible evidence owners apply only bounded factual/document corrections; re-freeze
   and re-review as required.
4. Only after evidence passes review, create a separate `EVIDENCE_DERIVED_OPTIONS` artifact.
   It may not perform new investigation or rewrite facts. Each option includes exact
   evidence/unknown IDs, consumer/decision effect, no-build/defer alternative, assumptions,
   dependencies, reversibility/rollback, estimate/confidence, affected owners, and external
   or Leo decisions. Mark `IMPLEMENTATION_AUTHORIZED: NO`.
5. Advisor performs final targeted audit and returns to Strategy.
