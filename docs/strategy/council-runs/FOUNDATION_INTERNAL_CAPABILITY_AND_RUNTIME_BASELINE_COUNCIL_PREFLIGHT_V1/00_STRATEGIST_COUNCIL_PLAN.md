## 1. Exact decision question

Is the proposed `FOUNDATION_INTERNAL_CAPABILITY_AND_RUNTIME_BASELINE_V1` investigation
the smallest sufficiently complete and safe way to determine how far Foundation has
progressed toward its ultimate mission, and to produce a reviewed current-state map plus
a bounded implementation proposal without beginning implementation?

If not, what exact corrections must be made to its purpose, capability coverage,
evidence model, runtime-safety envelope, actor composition, closure conditions, or
Founder-decision routing before an exact bilingual Advisor instruction may be returned
to Leo for approval?

## 3. Foundation ultimate mission used for this preflight

The proposed baseline evaluates Foundation as the shared trusted knowledge, evidence,
judgment, safety, retrieval, and AI core for SIASIU, Cosmile, and future services.

The long-term capability direction is:

```text
authoritative sources
-> provenance and versioning
-> atomic normalization and relationships
-> keyword/vector/graph/hybrid retrieval
-> evidence-grounded reasoning, judgment, and safety
-> stable downstream API results with degraded behavior
-> governed feedback and candidate-knowledge generation
-> review, versioned promotion, correction, retraction, and rollback
```

Products should ultimately be representable at decision-useful atomic granularity for
future person-product matching. This includes identity, brand, line, variant,
formulation, ingredient, ingredient role and amount where authoritative, claim,
evidence, warning, contraindication, allergy, safety, concern, skin type, use purpose,
usage, compatibility, locale, market, rights, provenance, version, correction,
retraction, and relationships.

This is a target-fitness standard for investigation. It is not authority to redesign
files, create a schema, process personal data, activate autonomous learning, or promote
new canonical knowledge.

The baseline is broader than an AI/search review. It must evaluate current Foundation
capabilities, their physical ownership, source state, buildability, tests, isolated
runtime behavior, downstream integration, operational dependencies, and current role
authority.

## 5. Required state separation

Every material capability must record separate states, not one ambiguous readiness enum:

```text
SOURCE_STATE
BUILD_STATE
TEST_STATE
RUNTIME_STATE
INTEGRATION_STATE
AUTHORITY_STATE
TARGET_MISSION_FIT
```

Each state must cite exact evidence and may use only bounded values defined by the later
Advisor mission. A capability with source and tests but no current runtime evidence must
not be called operational. Historical physical location must not be treated as current
implementation authority.

## 6. Required capability coverage

The proposed Advisor baseline must cover at least:

1. repository, branch, history, package, runtime, and ownership surfaces;
2. canonical Vault and knowledge-source structure;
3. provenance, rights evidence, versions, correction, retraction, and stewardship;
4. atomic product representation and relationship readiness;
5. normalization, identifiers, duplication, semantics, and freshness;
6. keyword, vector, graph, hybrid retrieval, embedding, index, and reranking;
7. knowledge-base, logic, judgment, evidence-composition, and safety behavior;
8. model/provider responsibility, prompts, routing, fallback, evaluation, latency, and cost;
9. Memory V1/V2/V3 source, tests, runtime, authority, and active/inactive boundaries;
10. Foundation API surfaces and degraded/fail-safe behavior;
11. SIASIU and Cosmile adapters and actual response consumption;
12. person-product matching readiness without processing real PII;
13. future governed learning readiness: intake, candidate generation, review, promotion,
    rollback, audit, and authority boundaries without activating Memory V3;
14. operational packaging, observability, restart, cleanup, and owner dependencies;
15. exact retain, repair, bounded-transfer, replace, retire, defer, and unverified
    recommendations.

## 7. Knowledge-use evidence chain

The baseline must not stop at file existence or record counts. Representative items must
be traced through:

```text
canonical source
-> provenance and version
-> normalization
-> index inclusion
-> runtime retrieval
-> reasoning or judgment use
-> response-level evidence use
-> downstream consumption where authorized
```

Missing links remain explicit and must not be inferred from adjacent layers.

## 8. Proposed isolated execution authority

Allowed only after Leo approves the later Advisor instruction:

- read-only repository and canonical-source inspection;
- isolated non-production build and existing tests;
- temporary API, retrieval, index, and synthetic-question execution;
- temporary local databases, indexes, caches, and runtime processes;
- approved non-production provider execution only when existing credential and network
  authority is explicitly verified;
- evidence capture that contains no secret, PII, customer data, or protected content;
- complete shutdown and cleanup verification.

Prohibited:

- product-code or tracked canonical-source changes;
- shared, staging, or production data and infrastructure;
- real PII, customer data, live flags, public exposure, or production deployment;
- new schema application or migration to shared resources;
- secret creation, disclosure, or unapproved provider calls;
- Memory V3 activation, sender/intake activation, autonomous promotion, or learning-loop activation;
- Foundation AI becoming a synchronous dependency of ordinary commerce;
- implementation or automatic follow-up work.

Temporary resources must be mission-owned, isolated, reproducible where practical, and
removed with explicit cleanup evidence. If an external provider cannot be used safely,
the state remains `PROVIDER_EXECUTION_NOT_RUN` or `UNVERIFIED` rather than being simulated
as official-provider evidence.

## 9. Repository and authority coverage

The later baseline must inspect both:

- the current Foundation repository and canonical sources;
- historical Foundation-related implementation physically located in
  `foundation-control`.

Historical code is evidence of physical location and prior behavior only. It does not
grant Control current Worker authority, move canonical ownership, authorize transfer,
or authorize deprecation. Any future bounded-transfer proposal must identify exact
source, target owner, dependency, compatibility, migration consequence, rollback, and
Founder decision.

## 12. Baseline closure and next-step boundary

The baseline ends after the reviewed current-state map and proposed bounded
implementation scope return to Strategy. No implementation follows automatically.

Strategy would then prepare an evidence-based recommendation and, because the resulting
scope may change ownership, data structure, AI behavior, security, privacy, or learning
direction, run a separate post-evidence Council challenge before returning a final
implementation recommendation to Leo.

## 14. Council preflight investigation depth

Allowed:

- read this frozen plan;
- read the frozen facts/assumptions/unknowns register;
- read the common mission brief and only the assigned role-specific prompt;
- challenge purpose, scope, evidence, safety, authority, actor allocation, closure, and
  unknown routing from the assigned lens.

Prohibited:

- inspect Foundation, SIASIU, Cosmile, foundation-control, or product repositories;
- perform the proposed baseline partially or fully;
- build, test, run APIs, create indexes, access DBs, call providers, or inspect secrets;
- research external law, policy, vendors, models, or current web facts;
- modify this plan or any repository;
- dispatch Advisor or Foundation Team actors;
- claim independent-review, legal, security-audit, or product-approval status.

## 15. Council rounds and termination

Round 1 is blind independent review. Round 2 is a Strategist-mediated cross-review after
all initial reports are frozen. Round 3 is a focused rebuttal only for material
disagreements. Rounds 4 and 5 may occur only when a difficult, decision-relevant issue
remains unstable and a further round can answer an exact question; they may not seek
artificial consensus or repeat arguments.

The Council ends when all material issues are classified, positions are stable, factual
unknowns are routed to non-Council resolution, and minority findings are preserved.
More than five rounds requires new Leo approval.

## 16. Strategist return boundary

After Council completion, Strategy will prepare:

- complete finding disposition;
- corrected investigation scope;
- exact bilingual Advisor instruction candidate;
- concise English and Korean Leo decision package.

Strategy returns these to Leo and stops. It must not dispatch the Advisor or start the
Foundation baseline until Leo explicitly approves the corrected bilingual instruction.
