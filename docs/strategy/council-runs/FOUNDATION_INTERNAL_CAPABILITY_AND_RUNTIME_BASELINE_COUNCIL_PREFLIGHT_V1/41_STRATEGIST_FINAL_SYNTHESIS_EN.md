## Final conclusion

The Foundation-wide baseline should be run, but not as a uniform deep audit of every
repository, retrieval modality, model, memory path, and downstream application. The
smallest safe method is:

1. a Foundation-wide shallow census of physical source, canonical authority, current
   consumers, capability state, and evidence availability;
2. an immutable scope and evidence checkpoint;
3. targeted isolated runtime traces only for questions capable of changing Leo's next
   Foundation priority decision;
4. conditional historical foundation-control and downstream inspection only for exact
   unresolved boundaries;
5. an independently reviewed current-state freeze;
6. separate evidence-derived options with no implementation authority.

This is the right next step because prior work did not establish current Foundation-wide
source/build/test/runtime/integration/authority/target-fit truth. It also prevents a repeat
of Memory V3-style platform expansion: a missing capability is not automatically a build
priority, and an ultimate-target gap is not product progress unless it serves an approved
current or near-term outcome.

## Decision contract to pin before dispatch

The later baseline must enable Leo to decide:

> Which current Foundation capabilities are trustworthy and reusable, which exact gaps
> deserve the next bounded implementation priority, and which capabilities should be
> retained, repaired, boundedly transferred, replaced, retired, deferred, or remain
> unverified so Foundation can provide an evidence-grounded AI/knowledge/judgment core to
> SIASIU and optional Cosmile AI surfaces without coupling ordinary commerce?

Approved near-term outcome envelope for evidence ranking:

- SIASIU evidence-grounded consultation and judgment consumption;
- Cosmile optional AI consultation and Foundation product-snapshot boundary, never the
  synchronous ordinary-commerce path;
- Foundation canonical product/brand/ingredient/claim/warning/safety/provenance services;
- atomic-product and governed-learning readiness only as future structural/control maps,
  not implementation or activation.

P1 must verify whether these consumers and paths actually exist, are active, partial,
dead, historical, or unverified. If no current or approved-near-term consumer justifies a
deep runtime slice, that slice returns `NO_CONSUMER_IDENTIFIED`, `DEFER`, or `HOLD`; target
fit alone does not create priority.

## Mandatory corrections before Advisor dispatch

1. Use one bounded P0-P4 mission with exact internal gates and material-escalation return.
2. Keep seven canonical axes: `SOURCE`, `BUILD`, `TEST`, `RUNTIME`, `INTEGRATION`,
   `AUTHORITY`, and `TARGET_FIT`.
3. Add applicable adjunct records for decision relevance, source authority/lineage,
   legal-policy dependencies, security-control effectiveness, AI behavior evaluation, and
   probe/evidence/cleanup.
4. Prohibit composite `READY`, `SAFE`, `COMPLIANT`, `OPERATIONAL`, or AI-readiness claims.
5. Use risk-tiered probe admission. Missing authority means not run/unverified; an actual
   breach or cleanup failure stops the affected phase.
6. Require a source-authority ledger before trusted-knowledge claims and preserve
   historical Control as physical evidence only.
7. Freeze inventory-derived trace classes, exact transformations, negative/degraded
   cases, expansion triggers, and path-limited claim ceilings before runtime work.
8. Treat atomic matching as `STRUCTURAL_READINESS_ONLY` with synthetic profiles and no
   schema/policy decision.
9. Treat learning as `SOURCE_AND_CONTROL_READINESS_ONLY`; keep Memory V3 and every
   promotion/persistence path inactive.
10. For selected stochastic AI cases, use 5 critical/3 routine repetitions as defaults,
    not universal minima or reliability evidence. Variance or failure narrows the claim.
11. Freeze facts and obtain independent evidence review before deriving options.
12. Preserve every factual unknown under its correct repository, authoritative-source,
    vendor, counsel, implementation-validation, or Leo-decision route.

## Recommended phase authority

One Leo approval should cover P0-P4 so routine internal gates do not create another
approval-writing loop. Automatic continuation is allowed only when the next phase stays
inside the exact repositories, actors, no-change/data/provider boundaries, decision
contract, and P0 estimate envelope.

Recommended material-deviation rule for Leo's approval:

```text
P0 must publish a finite engineering-workday and elapsed-time range with confidence.
Return to Leo if projected completion later exceeds the P0 upper bound by more than
max(1 working day, 25%), or if any new repository, actor class, provider authority,
data class, product/canonical write, architecture/ownership decision, or risk class is
required.
```

If Leo does not approve that rule, authority should stop after P0/P1 and require a second
approval before runtime/deepening.

  and final claim ceilings.
- Designer: not selected for this factual baseline.

Advisor must live-verify approved runtime bindings. Strategy recommends maximum reasoning
effort for the primary Foundation Worker and Independent Reviewer and at least xhigh for
targeted actors, but the Advisor owns binding verification and must not change approved
models without Leo approval.

## Evidence and claim ceiling

The baseline may report only exact pinned observations and bounded pass/fail results for
the environments, data/index snapshots, configurations, routes, providers, prompts,
scenarios, and repetitions actually inspected. It may not claim:

- production or operational readiness;
- general AI capability or safety;
- corpus completeness;
- supported-language quality beyond evaluated cases;
- real-person matching readiness;
- autonomous-learning readiness;
- Legal, security, privacy, or regulatory approval;
- provider suitability beyond authorized observed evidence;
- implementation approval.

## Decisions deferred until evidence returns

- exact implementation priority and WorkUnits;
- any retain/repair/transfer/replace/retire decision;
- canonical product schema or matching taxonomy;
- real-person attributes, consent, retention, deletion, or privacy operating model;
- model/provider/routing/retrieval/index redesign;
- ownership migration from foundation-control;
- Memory V3 or governed-learning activation;
- SIASIU/Cosmile integration changes;
- production, live, or public exposure.
