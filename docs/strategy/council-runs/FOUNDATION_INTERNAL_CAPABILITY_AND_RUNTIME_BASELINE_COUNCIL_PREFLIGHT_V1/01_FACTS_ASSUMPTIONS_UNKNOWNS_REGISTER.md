## Register rules

- Facts below are limited to explicit Leo direction, current canonical role authority
  already admitted by Strategy, and the bounded claims of previously reviewed work.
- This preflight does not claim current repository or runtime truth.
- Assumptions remain assumptions until the later approved baseline supplies evidence.
- Unknowns may not be silently converted into facts or assumptions.
- New unknowns proposed by a Challenger must contain every field used below and must be
  recorded separately by Strategy; this frozen register is not edited.
- `RESOLVED_BY_COUNCIL_ANALYSIS` applies only to strategy, scope, architecture,
  prioritization, and risk reasoning. It may not close a factual unknown.

## Verified direction and authority facts

### F-001 — Foundation ultimate direction

Leo directs Foundation to become the shared AI, knowledge, evidence, judgment, and safety
core for SIASIU, Cosmile, and future services.

Evidence: explicit Leo direction in the current Strategy thread.

### F-002 — Atomic product and future matching direction

Leo intends product knowledge to be decomposed and recorded at atomic granularity so it
can later support person-product matching.

Evidence: explicit Leo direction in the current Strategy thread.

### F-003 — Future learning direction is not current activation authority

Leo expects Foundation eventually to learn from new knowledge. Current authority does
not activate autonomous canonical promotion, Memory V3, sender, intake, or a learning
runtime in this baseline.

Evidence: current Leo direction plus the standing Memory V3 pause.

### F-004 — Investigation-only boundary

The proposed next mission is investigation and verification only. Product code,
canonical Vault, schema, shared data, and implementation may not be changed.

Evidence: Leo-approved baseline corrections in the current Strategy thread.

### F-005 — Isolated non-production verification permission

The proposed later mission may perform isolated non-production build, test, API,
retrieval, index, and synthetic-question execution, subject to explicit safety and
cleanup boundaries.

Evidence: Leo-approved baseline corrections in the current Strategy thread.

### F-006 — Current repositories and historical implementation

The proposed baseline must inspect the current Foundation repository and historical
Foundation-related implementation in `foundation-control`. Historical code does not gain
current implementation authority automatically.

Evidence: Leo-approved baseline corrections and current Team role boundary.

### F-007 — Required state separation

Source, build, test, runtime, integration, and authority states must remain separate.

Evidence: Leo-approved baseline corrections.

### F-008 — Vault evidence requirement

The baseline must verify canonical sources, provenance, versions, index inclusion,
runtime consumption, and response-level use rather than file existence or record counts
alone.

Evidence: Leo-approved baseline corrections.

### F-009 — Standing exclusions

Memory V3 remains paused. Cosmile Golden Commerce remains unchanged. Foundation must not
become a synchronous dependency of ordinary commerce.

Evidence: explicit Leo direction and standing Founder decisions.

### F-010 — Prior investigations were differently scoped

Previous reviewed work established a commercial baseline, Foundation product snapshot
work for Cosmile O1, and bounded Memory V3 pre-runtime evidence. It did not claim a
Foundation-wide internal capability and runtime baseline across all surfaces proposed
here.

Evidence: reviewed mission scopes already returned through Advisor and Strategy; this
preflight does not reuse them as current runtime proof.

### F-011 — Current Council return boundary

Leo explicitly authorized this Council preflight and requires Strategy to return the
Challenger result for confirmation before any Advisor baseline dispatch.

Evidence: current Leo instruction.

## Explicit assumptions to challenge

### A-001 — A bounded Foundation-wide baseline is feasible

The plan assumes broad capability coverage can remain decision-useful without becoming
an exhaustive code-quality, security, Legal, or architecture-redesign program.

### A-002 — Isolated runtime verification is feasible without product changes

The plan assumes existing scripts, dependencies, and temporary resources can support
meaningful execution without modifying tracked product code or canonical knowledge.

### A-003 — Atomic-product readiness can be assessed before a final schema decision

The plan assumes current file semantics, identifiers, provenance, and relationships can
be evaluated against a target capability without prematurely designing or approving the
future canonical schema.

### A-004 — Representative traces can expose system truth

The plan assumes carefully selected knowledge-to-response traces and synthetic questions
can test every material layer without requiring an exhaustive corpus evaluation.

### A-005 — Historical code can be evaluated without silent ownership transfer

The plan assumes Control can provide targeted evidence while current Worker and canonical
ownership boundaries remain intact.

### A-006 — Downstream verification can be targeted

The plan assumes SIASIU and Cosmile adapter checks can prove boundary and consumption
states without expanding into complete application audits or colliding with active work.

### A-007 — Future learning readiness can be evaluated without activation

The plan assumes candidate-generation, review, promotion, rollback, and governance
readiness can be mapped from source and isolated evidence while Memory V3 remains paused.
