## Purpose and primary decision question

Determine how much of Foundation itself is actually implemented, buildable, tested,
runtime-demonstrated, integrated, and governed today.

Return evidence showing:

- which Foundation capabilities truly work;
- which are source-only, mock, shadow, partial, historical, dead, blocked, or unverified;
- which canonical knowledge, retrieval, model, judgment, safety, evidence, API, runtime,
  and memory components exist;
- how far Foundation has progressed toward its own current canonical mission;
- what should be retained, repaired, boundedly transferred, replaced, retired, deferred,
  or investigated further;
- what one evidence-derived next Foundation priority, no-build/defer result, further-
  evidence requirement, or hold result should later be considered.

## Foundation-only outcome envelope

Assess Foundation itself:

- canonical mission, knowledge, provenance, versions, rights status, correction,
  retraction, and stewardship;
- product, brand, ingredient, claim, warning, safety, evidence, and source structure;
- atomic product representation at `STRUCTURAL_READINESS_ONLY`;
- lexical, vector, graph, hybrid, embedding, index, retrieval, and reranking capabilities;
- deterministic logic, model/provider responsibility, routing, prompts/context, fallback,
  judgment, verdict, safety, and evidence composition;
- API packaging, runtime entry points, failure/degraded behavior, restart, and observability;
- internal memory and learning-control state without activation;
- physical, implementation, runtime, canonical, steward, contract, and decision authority;
- operational and governance readiness at exact evidence ceilings.

Do not use an invented future architecture as the grading target. P0/P1 must pin the
current canonical Foundation mission and distinguish:

```text
CURRENT_OPERATIVE_REQUIREMENT
CURRENT_PRINCIPLE
FUTURE_DIRECTION
SUPERSEDED
CONFLICT
UNVERIFIED
```

Only current operative requirements and current principles may justify P2 depth. Future
direction appears only under bounded `TARGET_FIT`.

## Explicit non-objectives and prohibitions

SIASIU development is not a current near-term objective. Cosmile AI development is not a
current objective. Neither may define the target architecture, consumer requirement,
evaluation success, or reason to deepen a Foundation path.

Prohibited:

- SIASIU implementation or feature design;
- Cosmile AI implementation or feature design;
- modification of Cosmile Golden Commerce;
- Foundation becoming a synchronous dependency of ordinary commerce;
- Foundation product-code or canonical-knowledge modification;
- schema or migration application;
- shared, staging, or production data/infrastructure;
- real PII, customer data, live flags, public exposure, or production deployment;
- secret creation, disclosure, or unapproved provider use;
- Memory V3, sender/intake, autonomous jobs, promotion, serving mutation, or canonical write;
- real-person matching, consent-policy, or recommendation-policy decision;
- Foundation implementation, ownership transfer, merge, deployment, risk acceptance, or
  automatic follow-up.

  confidence, evidence volume, Reviewer/correction allowance, external dependencies,
  external cost ceiling, critical path, and collision state.


## P2 — Selected isolated Foundation verification

Execute only P1-selected slices in mission-owned isolated non-production resources.

Risk-tiered evidence templates:

- Tier 0: deterministic read-only pin/method/output/failure/owner;
- Tier 1: isolated local execution plus expected writes/resources/abort/cleanup;
- Tier 2: network, provider, credential, sensitive, or mutation-capable execution with the
  full deny-by-default record.

Allowed when admitted: existing build/tests, local API, retrieval/index, synthetic questions,

`PROVIDER_EXECUTION_NOT_RUN`. Boundary breach, secret/PII exposure, shared/production
contact, public listener, canonical mutation, Memory/promotion activation, cost overrun, or
cleanup failure stops the affected phase and returns `HOLD` with preserved evidence.

## P4 — Evidence freeze, Independent Review, and priority result

1. Freeze one content-addressed evidence index and artifact manifest.
2. Independent Reviewer fully examines admission, prohibited-boundary, authority, failed/
   blocked, claim-ceiling, candidate-supporting, and cleanup evidence, plus a risk-based
   reproducibility sample of passing rows.
3. Permit one bounded factual mission-evidence correction/re-freeze/re-review cycle inside
   the accepted envelope. Further material change, repeated reproduction failure, or
   evidence-set expansion returns `HOLD` or `UNVERIFIED`; it does not loop.
4. Advisor performs final targeted audit and returns the reviewed evidence.
5. Produce exactly one `P4_PRIORITY_RESULT`:

```text
BOUNDED_IMPLEMENTATION_CANDIDATE
NO_BUILD_OR_DEFER
FURTHER_EVIDENCE_REQUIRED
HOLD
```

If a candidate is supported, it must state exact evidence/unknown IDs, current outcome and
acceptance measure, why now, scope/exclusions, `PROPOSED_OWNER`, dependencies, reversibility,
rollback, tests, estimate/confidence, comparison with no build/defer and the next-best
bounded alternative, consequence of waiting, and unresolved Leo/external decisions.

P4 may not decide canonical identity, source, owner, schema, identifier, merge, migration,
correction, retraction, rights, stewardship, ownership, API, AI behavior, canonical write,
or ordinary-commerce dependency. Such elements are
`PROPOSED_PENDING_POST_EVIDENCE_COUNCIL_AND_LEO_DECISION`. Factual corrections apply only
to mission evidence artifacts, never product code or canonical knowledge.

Every P4 result must state:


After Advisor closure, return to Strategy. Strategy routes the evidence and P4 result
through the previously approved post-evidence Council challenge, performs final synthesis,
and returns to Leo. No implementation or next mission begins automatically.
