## 3. Findings by severity

**F-1 (REQUIRED — mission-conformance, Domain §6.1/§13 + UI §5): required observable-state contract not demonstrably satisfied.**
The mission requires these 16 user-facing WorkUnit states: QUEUED, READY, DISPATCHING, READING, WORKING, TESTING, WRITING_RESULT, RETURNING_RESULT, REVIEWING, NEEDS_PATCH, WAITING_DEPENDENCY, WAITING_LEO, BLOCKED, COMPLETED, FAILED, CANCELLED. The candidate's two axes cover: QUEUED/READY/TESTING/NEEDS_PATCH/WAITING_DEPENDENCY/WAITING_LEO/BLOCKED/COMPLETED/FAILED/CANCELLED as primary states; READING/WORKING/REVIEWING(as REVIEW)/RETURNING_RESULT(as RESULT_RETURN) on the activity axis; DISPATCHING via DISPATCHED(primary)+DELIVERY(activity, though the DELIVERY trigger text is Advisor-notification-scoped and never says it also represents WorkUnit dispatch-in-progress). **`WRITING_RESULT` appears on neither axis**: the WORKING activity is triggered by RUNNING-entry, and RESULT_RETURN fires only after `RESULT_REPORTED` plus a verified result artifact — the "actor is writing its result" observable is unrepresentable as designed. Additionally, the candidate contains **no mapping table and no rationale** connecting the 16 required names to its model — and the mission text forbids accepting a renamed/lost state without evidence and rationale. Verdict on the architecture question: the two-axis model is SOUND and should be kept (a flat 16-name machine would conflate durable audit states with ephemeral observables — several required names such as READING/WORKING are inherently activity-kind, and the two-axis split makes both event-sourced and auditable). The defect is conformance documentation and one missing observable, not the architecture. Required patch (doc-level, Worker's choice of exact form): add an explicit required-state conformance table (all 16 names -> primary state and/or activity + rationale per rename), and represent WRITING_RESULT explicitly (e.g., an added activity value between WORKING and RESULT_RETURN, or a documented mapping with rationale); if DELIVERY is also the dispatch-in-progress representation, say so in its trigger row.

**F-2 (REQUIRED — Domain §7.2/§7.3): blocker type vocabulary and required blocker fields, and alert-kind vocabulary, are absent.**
The contract fully defines blocker and alert lifecycles, but: no typed blocker-kind enum exists; no blocker payload schema declares the required fields (exact reason, safe default, owner, next action, evidence references) — the mission and review brief name these fields explicitly, and the UI already renders "blocker reason, route, age/freshness" (UI §5 BLOCKED row) from fields the contract never declares. Alert kinds are similarly narrated (UI §2/§13: stale-evidence/security/transport/recovery) but never enumerated as a typed vocabulary, even though NotificationSink derives deterministic IDs "from kind" (Integration §10). Without these, the Batch A implementer must invent required observables — precisely the product-policy invention this design is supposed to prevent. Required patch (doc-level): add `BlockerOpened` payload schema (typed blockerKind vocabulary + reason, safeDefault, owner, nextAction, evidenceRefs, resume destination fields) and a typed alertKind vocabulary to the contract, with the UI/notification documents referencing them.

**F-3 (RECOMMENDED — UI doc): Korean user-facing label specification absent.**
Only the imported `labelKo` (initiative/package) and a text-expansion test rule exist. The required Korean hierarchy labels, state labels, alert actions, and blocker-reason display vocabulary are unspecified — an implementer would invent user-facing Korean vocabulary. Recommended patch: a Korean label table (or an explicit declaration that all user-facing labels come from a named, to-be-approved i18n source with the manifest `labelKo` rule generalized), so Batch B/C need no invention.

**Observations (INFO, no action required):** GPT copy-package fields are specified as input classes rather than a field-by-field schema — acceptable at design level; Batch D should pin the exact schema. Rate limits are correctly labeled candidate constants for Batch E review. The static-launcher delivery design (Integration §4.1) is a notably strong choice — it keeps even active-mode tmux input to a fixed committed launcher, exactly matching V2 12A exact-launcher semantics.
