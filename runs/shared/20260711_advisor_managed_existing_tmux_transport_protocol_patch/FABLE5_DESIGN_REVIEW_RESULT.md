## 2. Design-review coverage (04_FABLE5_REVIEW_BRIEF Pass-1 items, answered explicitly)

1. **Transport authority narrower than Worker/Control/Reviewer? YES.** `§12A`: "does not add an actor and does not transfer Worker, Control, Reviewer, risk-acceptance, or final-approval authority"; TRANSPORT_PROTOCOL `§12` "Authority That Never Transfers" enumerates the same, including no `PASS_WITH_RISK` acceptance and no next-mission selection. Transport = routing/evidence only; `§12A` closes with "If transport execution would require judgment rather than exact routing or evidence collection, STOP" — the same judgment boundary as the pre-existing Hermes `§12`.
2. **Activation truly fail-closed, manual routing active now? YES.** ACTIVATION_STATE: `MODE_STATUS: NOT_ACTIVE`, `KILL_SWITCH: ENGAGED`, `MANUAL_ROUTING_FALLBACK: ACTIVE`, `FINAL_ACTIVATION_RECORD: NOT_PRESENT`, plus the explicit fail-closed rule ("any missing, ambiguous, stale, or conflicting state means NOT_ACTIVE"). `V2 §20` requires seven gates (patch, two Fable5 PASS verdicts, six session reloads, Advisor final audit PASS, separate Leo/GPT approval, committed activation record). README: file presence does not activate. Template: "not an activation record" + fail-closed unless every field verifiable. Config authorship/review/reload is explicitly not activation permission.
3. **Misdelivery controlled? YES.** Layered: registry is a locator not proof; live pre-dispatch revalidation of session/window/pane/workspace/process/actor/readiness; pane IDs distrusted after mismatch (`HOLD_REGISTRY_MISMATCH`, no guessing/renaming/clearing); launcher path+commit+blob+dual SHA-256 with worktree-vs-blob STOP; unique named buffer, exact single-pane target, final submit key only; no wildcard/multi-pane/broadcast; Advisor's own session is registered as "never a target". Unregistered tmux sessions are structurally out of reach (registry whitelist).
4. **Role drift, concurrency, branch collision, blindness controlled? YES.** Serial by default; mandatory serialization of dependent steps, Worker-then-Reviewer, patch-then-re-review, same-repository/same-branch writes (explicitly including the shared `foundation-docs` branch — the realistic collision path), result-consuming tasks, and independence-contaminating tasks. Parallel dispatch requires all eight conditions including isolation proof in a dispatch matrix and "no actor may read another actor's first-pass result" (canonicalizing the Package 1A blind-independence lesson). Pre-dispatch ledger scan of active `SENT/RUNNING` entries gates lock clearance (`HOLD_DEPENDENCY_OR_WRITE_CONFLICT`), and ledger completion may not be assumed until durable result and Git evidence reconcile. Role drift is a named kill-switch trigger.
5. **Unexpected interactions cannot be auto-approved? YES.** `§8` allows responses only to predicted, in-mission, non-sensitive, non-expansive interactions; auto-answering approval/auth/privilege/DB/secret/protected-branch/production/live/destructive-Git/force-push/scope prompts is forbidden; the required behavior is kill-switch + return to Leo/GPT. `HOLD_INTERACTIVE_OR_AUTH_RISK` exists as a preflight verdict.
6. **Timeout/stall cannot silently interrupt or broaden? YES.** Soft stall = inspection only. Hard timeout engages the TRANSPORT kill switch and manual fallback; it "does not automatically terminate the role process"; `Ctrl-C`/termination/reset/retry require separate mission authority. Broadening is impossible from transport: follow-up routing (`§10`) is limited to already-authorized steps.
7. **Kill switch and fallback immediate and unambiguous? YES.** Dedicated file with engage-immediately trigger list (mismatch, wrong-target, broadcast detection, unexpected sensitive prompts, checksum failure, isolation failure, role drift, unresolvable evidence), six-step effect (no new input, no retry/approval/Ctrl-C, ledger STOPPED marking, evidence capture, canonical return, manual-only until revalidated), re-enable conditions, and Leo/GPT-only initial disengage. `§11` lists nine conditions where manual routing is mandatory — manual fallback is always available and is the current default.
8. **Result collection distrusts pane summaries? YES.** `§9`: pane output is progress signal only; Advisor must directly verify result artifact, pointer, allowed-vs-actual changed files, diff/test evidence, commit/branch/upstream/push/ancestry, runtime-unchanged evidence, and review verdict/coverage/session separation; "missing or contradictory durable evidence prevents progression even if the pane reports success." This encodes reported-vs-actual discipline at the transport layer.
9. **Final approval and escalation remain Leo/GPT-owned? YES.** `§12A` tail, `§12`, `§20` gate 6, kill-switch initial disengage, and `§10` STOP-and-return for anything requiring new authority.
10. **No hidden runtime or product authorization? YES.** `§20`: "No product mission is authorized by this maintenance decision"; intake forbidden list; the supersession clause (`§16` addition) is tightly scoped — it supersedes ONLY older manual-paste wording, ONLY after final activation, ONLY for transport satisfying `§12A`, and explicitly not actor separation, mission scope, independent review, risk acceptance, final approval, or DB/secret/protected-branch/production/live restrictions.

## 3. Adversarial design observations (INFO — recorded, not blocking)

- **O-1 Preflight-to-paste freshness (TOCTOU).** `§3` requires all checks to pass "before input is sent" but does not state a maximum staleness between the live checks and the paste. The failure mode (target process exits to a shell between check and paste) is bounded by multiple factors — process match is a live check, `paste-buffer -p` delivers one bracketed paste (bracket-aware shells do not auto-execute pasted newlines), only a single final `Enter` is sent, and wrong-target discovery is an immediate kill-switch trigger — but an explicit freshness rule (e.g., re-capture/verify in the same immediate command sequence as the paste) would close the residual window. Recommend carrying this as a line item in the final-activation record or a one-sentence protocol clarification at the next touch. Not blocking: the current text already reads naturally as check-then-immediately-send.
- **O-2 Role evidence is a soft signal.** "Latest pane record declares ACTOR" is screen-content-based and could in principle be satisfied by displayed text rather than session identity. It is one factor among five (session name, session ID, workspace, process, marker) plus the launcher's own TARGET_ACTOR header and each role session's reload-confirmed understanding (`§18` addendum) that it executes only its own actor's prompts. Acceptable as designed.
- **O-3 Ledger-as-lock assumes a single Advisor.** The dispatch-ledger scan serializes correctly only if exactly one Advisor session dispatches. The registry contains exactly one Advisor row and Advisor is "never a target"; V2 gives Advisor field management to one actor. Acceptable; if a second Advisor session ever exists, the protocol needs a stronger lock.

The three observations are informational design notes whose worst cases are bounded by existing triggers and gates; none requires Leo/GPT risk acceptance before the next gate (reloads + final activation), and O-1 can be absorbed as a wording clarification at any next touch.
