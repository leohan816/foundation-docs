## 0. 한 줄 철학
**"검색엔진을 똑똑하게 만들기 전에, 검색엔진에 넣는 질문을 똑똑하게 만들어라."**
검색 품질은 3단계로 결정된다 — ①무엇을 검색할지 이해(Query) ②올바른 후보를 충분히 가져옴(Recall) ③좋은 순서로 재정렬(Rerank). 가장 큰 레버는 **①**이다.

## 0.5 ★ 정확도 우선 원칙 (SIASIU의 생명, Leo 2026-06-19)
**SIASIU은 속도보다 정확도가 우선이다.** 속도도 중요하지만, *둘이 충돌하면 정확도가 이긴다.* 이유: 피부·건강·성분·시술 상담에서 **틀린 답 한 번 = 신뢰의 죽음**(그리고 안전 사고). 그래서 봄(음성·일상, Fast 기본)과 정책 기본값이 다르다:

1. **상담은 기본 Careful Path.** 인사·단순조회 같은 *명백히 가벼운* 질의만 Fast. 실질 상담(증상·제품·성분·시술)은 **기본적으로 리랭커 ON + 엄격 grounding**. (봄은 Fast 기본 → SIASIU은 Careful 기본)
2. **리랭커 트리거 임계를 낮춘다(켜는 쪽으로 기울임).** 애매하면 끄지 말고 켠다. latency를 정확도와 바꾸지 않는다 — 단 Query Massage가 대부분을 빠르게 유지하므로 비용은 제한적.
3. **Grounding "모름"은 폴백이 아니라 기능이다.** 근거가 약하면 *그럴듯한 빠른 답 대신* "확인이 필요해요 / 지금은 단정 못 해요 / 한 가지만 더 여쭤볼게요"를 **자신 있게** 낸다. 헌법(행동진실)과 정합.
4. **메모리 정합 = 정확도의 일부.** 사용자의 회피성분·과거반응·퍼스널컬러와 **모순되는 추천은 틀린 답**이다(빠르고 그럴듯해도). 메모리 채널을 항상 대조.
5. **고위험은 한 겹 더.** 의료·농도·시술·임산부/아동은 Careful + 추가 검증(원문 대조) + 단정 금지. 느려도 맞게.
6. **정확도 예산.** 중요한 상담 답은 수 초의 지연을 허용한다(사용자에게 "정확히 확인 중"을 보여주며). 가벼운 질의는 여전히 빠르게.

> 한 줄: **"빠르지만 틀릴 바엔, 조금 느려도 맞거나 정직하게 모른다고 한다."**

## 1. 배경 — BOM 3라운드 벤치가 증명한 것 (근거)
> 원자료·하네스: `~/Project/BOM/설계문서/검색품질_AB벤치_2026-06-19.md`, `검색품질_다국어_CLIR벤치_2026-06-19.md`, `검색품질_QueryMassage_2026-06-19.md`. (숫자 중복 최소화 — 헤드라인만)

- **① 어휘 vs 의미:** 어휘검색만으론 부족(일상어·다국어·개념 질의에 약함). 의미 단독도 100% 아님. **A∪B′ = 22/22 → 융합이 답.**
- **② 영어 리랭커는 해로움**(한국어 정답을 밀어냄, OFF 시 hit 0.82→0.86). **다국어 리랭커**(bge-reranker-v2-m3)는 전 언어 개선하지만 **latency 3.4초**가 발목.
- **③ Query Massage = 게임체인저:** 질문을 한국어 정본 검색어로 미리 보강하니 —
  - **latency** 3.4초 → ~0.1초 (VI 0.86 @ 41ms vs 원문+리랭커 0.64 @ 3.6초, **89배 빠르고 품질 더 높음**)
  - **교차언어 격차 소멸**: 마사지+리랭커(D2) 4언어 전부 **0.91** (KO=EN=ZH=VI)
  - **일반화**: 측정 안 한 id/th/**ar(아랍·다른 문자·RTL)**도 D2 **0.88 = 한국어급**
- **④ 변하지 않는 한 가지:** 리콜 갭(L2 마그네슘·C5 코로나후유증)은 **모든 모드에서 0** — 정답 문서가 색인에 *아예 없어서*. 리랭커도 마사지도 못 고침. → **큐레이션 키워드 채널만이 답.**

핵심 통찰: **리랭커는 후보 안에서 재정렬만 한다.** 정답이 후보에 없으면 못 찾는다.
→ **정밀 브레이크(리랭커) ≠ 리콜 구조대(큐레이션 채널 + 정직한 "모름").** 둘은 다른 부품.

## 2. 확정 파이프라인
```
사용자 질문 (어느 언어든)
  → 언어 감지 (detect_lang — 문자비율 결정론, ko/en/zh/그외)
  → ★ Query Massage  (의미 보강 검색 패킷 = 한국어 정본 + 핵심 원어 + risk flags)
  → ★ CLIR 양방향  (질의 언어 → 나머지 자료 언어 전체(ko/en/zh)로 LLM 번역 → 언어별 변형 질의)
  → Hybrid Search (변형별 검색 = 리랭크 OFF)
       ├ A: 큐레이션 키워드/BM25 검색  (리콜 구조대 — 색인된 제목·태그·검색키워드)
       ├ B: 벡터 의미검색 (다국어 e5)
       ├ G: 관계 그래프 확장·부스트 (성분→제품 1~2홉)
       └ M: 사용자 메모리 검색 (회피성분·과거반응·퍼스널컬러)
  → Fusion (RRF — 언어별·채널별 결과 병합)
  → [조건부] 다국어 리랭커(bge)  ← 기본 OFF, "조심 신호"에만 ON (§4)
       · ★병합 후 원문 질의로 1회만 리랭크 (변형별 중복 리랭크 제거)
  → Grounding Gate  (근거 약하면 안 싣고, 없으면 정직하게 "모름")
  → 답변 (출처 동반)
```

**★ consult→ssbrain KB 배선 (kb_search, 2026-06-23):** 상담(consult) 답변을 만들기 *전에* 위 엔진으로 **KB를 검색해 근거를 시스템 프롬프트에 우선 주입**한다 (이전엔 하드코딩 `SEED_KB` 6개에 의존). LLM이 그라운딩된 근거 위에서 답하므로 봄 라이브(BM25 단독)보다 완전하다 — 어휘+벡터+그래프+조건부 리랭커가 함께 후보를 만든다.

## 3. Query Massage — 질문을 검색가능한 의미 단위로
**번역이 아니다.** 사용자의 말을 "검색엔진이 잘 먹는 의미 단위 + 한국어 정본 용어"로 바꾼다. 모든 언어를 **한국어 검색 공간(0.91)으로 깔때기처럼 모은다** → 언어별 검색엔진 불필요, 새 언어 추가 = 검색 엔지니어링 0.

**검색 패킷 예 (뷰티):**
```json
{
  "original_query": "Da tôi đang bị kích ứng nhưng tôi muốn tìm cushion.",
  "original_language": "vi",
  "intent": ["color_cosmetic_recommendation", "sensitive_skin_risk"],
  "search_terms_ko": ["피부 자극","민감 피부","피부 장벽","쿠션 파운데이션","저자극 베이스 메이크업","패치 테스트"],
  "bridge_terms_en": ["skin irritation","sensitive skin","barrier damage","cushion foundation"],
  "risk_flags": ["active_irritation","avoid_aggressive_recommendation"],
  "product_category": ["cushion","base_makeup"],
  "recommendation_policy": "신중 추천 · 제품은 마지막 · 패치테스트 포함",
  "search_channels": ["keyword","vector","curated_keyword","graph","memory"]
}
```

**구현: 결정론 우선 + LLM 보조** (LLM이 매번 다 만들면 또 다른 latency)
```
Query Massage = 결정론 glossary/bridge  (한↔영↔중↔… 용어집, 우리가 소유)
              + 도메인 그래프 확장
              + LLM structured parser (DeepSeek 등 — 의미 구조화에 적합)
              + safety/risk classifier
```
> SIASIU은 DeepSeek 같은 저렴·고성능 모델을 쓸 가능성 높음 → **답변 생성뿐 아니라 "질문 의미 구조화"에 활용**. (단 최종은 우리 glossary/bridge 우선.)

### 3.1 ★ CLIR 양방향 (2026-06-23) — "번역된 지식 저장 말고, 번역된 질문으로 찾아간다"
자료(KB)는 여러 언어(ko/en/zh)로 섞여 있다. **자료를 번역해 저장하지 않는다**(원본 보존·중복 폭발 방지 — `clean-not-compress`와 정합). 대신 **질문을 자료 언어로 번역해 찾아간다:**
1. `detect_lang`(문자비율 결정론 — 휴리스틱 아님)으로 **질의 언어 감지**.
2. 질의를 **나머지 자료 언어 전체(ko/en/zh)**로 LLM 번역 → 언어별 변형 질의 생성.
3. 언어별로 검색(어휘+벡터+그래프) → **RRF로 병합**.

→ 어느 언어 질의든(측정 외 **아랍어 포함**) 우리 자료 언어로 번역돼 **모든 언어 코퍼스에 도달**한다. 이전 단방향(ko→en/zh 보강)을 **양방향**(질의 언어 ↔ 자료 언어 전부)으로 확장한 것. §0 철학("같은 의미 정본으로 수렴")은 유지하되, *수렴 방향이 한 점(한국어)에서 자료 언어 집합으로* 일반화됐다.

## 4. 리랭커 = 조건부 정밀 검증 (기본 OFF)
리랭커는 기본 엔진이 아니라 **"우리가 질문을 제대로 이해했는지 다시 확인하는 장치"**. 평소 꺼두고 **조심 신호**에만 켠다:
1. **검색 confidence 낮음** (top-1과 top-5 점수차 작음 · 주제 흩어짐 · grounding "근거 약함")
2. **사용자 정정** ("그 뜻 아냐", "다른 자료 없나", "방금 틀린 듯")
3. **동일 세션 2회 실패** → 강제 리랭커 모드 + "방금 의미를 잘못 잡았을 수 있어 다시 확인할게요"
4. **고위험 도메인** (피부 자극·영양제·피부과·시술·임산부/아동/질환 — SIASIU / 건강·기능의학 — BOM)
5. **교차언어 불확실** (저자원 언어에서 자신 없을 때 — 실측: id·ar S8을 리랭커가 0.75→0.88 복구)
6. **유료/프리미엄** ("늦더라도 더 정확히 확인" = 상품 가치)

**★ 리랭커는 원문 질의로 재정렬한다** (마사지된·번역된 질의가 아니라). 마사지가 헛짚었으면 여기서 잡힘 = "마사지 오류 보정"의 실제 메커니즘.
**★ 변형별 검색은 리랭크 OFF — 병합(RRF) 후 원문 질의로 *1회만* 리랭크한다** (2026-06-23). CLIR 양방향으로 언어별 변형 질의가 여러 개여도, 각각 리랭크하면 *중복 리랭크 + latency 폭발*. 그래서 변형 검색은 리랭크를 끄고, **모은 뒤 원문 질의로 단 한 번** 리랭크한다(모델: bge-reranker-v2-m3).
```
Fast Path:          rerank 없음 (massage 검색어로 충분)
Careful/Recovery:   변형별 검색(rerank OFF) → RRF 병합 → rerank(원문 질의, 후보 8~12개) 1회   ← 마사지 오류·애매·위험을 정밀 검증
timeout 2.5~3.5s → 초과 시 Fast Path 결과 + "불확실" 고지
```

## 5. 리콜 갭 = 큐레이션 채널 + 정직한 "모름"
- **리랭커·마사지는 리콜 갭을 못 고친다**(L2·C5 영구 0/4 = 증거). 정답이 후보에 없으면 끝.
- **유일한 해법:** ① **큐레이션 키워드 채널(A)**을 융합에 추가(제목·태그·검색키워드 색인이 L2를 찾음) ② 그래도 없으면 **grounding gate가 정직하게 "확인 못 했어요"** (헌법 행동진실).
- **고위험 상담의 진짜 안전 = "틀린 답 잘 정렬"이 아니라 "없으면 없다고 말하기".**

### 5.1 ★ 회피성분 결정론 배제 (2026-06-23) — 메모리 정합을 검색 단계에서 강제
사용자의 회피성분과 모순되는 추천은 §0.5-4 기준 *틀린 답*이다. 이를 리랭커 점수에 맡기지 않고 **결정론으로 배제**한다(휴리스틱 아님 — 정확한 그래프 규칙):
```
avoid_list(메모리) → bridge 원자(ing:<slug> 정본) → 그래프 related(성분→제품) → 그 성분 담은 제품 doc를 검색결과에서 제외
```
- `user_id`를 검색에 전달해 해당 사용자 컨텍스트로 필터링한다.
- 성분 식별은 **정본 atom**(`설계문서/SIASIU_성분레지스트리_설계서.md` §1.5)으로 — 별칭·표기 변형이 같은 원자로 수렴해야 그래프가 누락 없이 매칭된다.

## 6. Fast / Careful / Recovery — 3경로 정책
> **SIASIU 기본값 = Careful** (정확도 우선, §0.5). 봄은 Fast 기본이지만 SIASIU 상담은 신뢰가 생명이라 기본을 Careful로 둔다.

| 경로 | 언제 | 구성 | 목표 |
|---|---|---|---|
| **Fast** | **명백히 가벼운 질의만** (인사·단순 조회·UI 안내) | Massage + 어휘+의미+큐레이션+메모리 + Grounding | 빠름(50~200ms) |
| **Careful** (★SIASIU 기본) | **실질 상담 전부** + 고위험·교차언어 불확실·유료 | Fast + 리랭커(원문 질의) + 더 엄격한 grounding + "모름" 강화 | 피부·건강·시술·제품의 신뢰성 |
| **Recovery** | 사용자 정정·2회 실패 | 이전 답 무효 검사 + Massage 재생성 + 리랭커 강제 + 재검증 | 틀렸을 때 빨리 인정하고 정확히 복구 |

```python
if user_correction_detected:            path = "recovery"; reranker = "force_on"
elif repeated_failure_count >= 2:        path = "recovery"; reranker = "force_on"
elif is_trivial_query:                   path = "fast";    reranker = "off"   # 인사·단순조회만
else:                                    path = "careful"; reranker = "on"    # ★상담 기본 = Careful
# high_risk(의료·농도·시술·임산부/아동)·교차언어 불확실·유료 → 항상 Careful + 추가 검증
# 켤 때: candidate=8–12, timeout=2.5–3.5s, 초과 시 Fast 결과 + 불확실 고지
```

## 7. 안전 (헌법) — 확장은 강력하지만 위험하다
- **진단명·병명 추정을 검색어에 넣지 않는다.** "피부 따가워요"→`아토피·스테로이드·피부염 치료`는 **과잉의료추론 = 헌법 위반**(효능단정·의료조언 금지). 확장은 **안전한 동의어 · 일상어→표준개념 · 언어 bridge**만.
- **확장어에 confidence + source 태깅:**
```json
{"term":"피부 장벽 약화","confidence":0.82,"source":"domain_glossary","type":"safe_expansion"}
{"term":"피부염","confidence":0.35,"source":"llm_hypothesis","type":"medical_hypothesis","search_allowed":false}
```
- 위험 추론은 `search_allowed:false`인 hypothesis로만 보관(검색에 안 씀).

- SIASIU은 **글로벌**(고객이 각자 언어로 피부·건강·컬러·제품 고민) → Query Massage가 **글로벌 CLIR의 핵심 레이어**.

## 8.5 의미 읽기의 5단계 — "찾기"는 해결, "읽고·답하기"가 다음 (정직)
"의미를 읽고 답한다"는 5단계다. 오늘 벤치(BomBrain)가 해결한 건 **앞 2개(찾기)**다.
| 단계 | 상태 |
|---|---|
| 1. 질문의 의미 읽기 (Query Massage) | ✅ 크게 해결 |
| 2. 맞는 문서 찾기 (의미검색+조건부 리랭커) | ✅ 크게 해결 (단 색인에 있어야 — 리콜 갭 L2) |
| 3. 문서 읽고 답 생성 (LLM 두뇌) | ⚠️ **다음 큰 미지수 — 아직 측정 안 함** |
| 4. 근거 약하면 "모름" (Grounding Gate) | ⚠️ 설계만, 구축·측정 미완 |
| 5. 라이브 연결 | ⚠️ 아직 (shadow) |

★ **가장 정직한 포인트:** 오늘 좋아진 건 **"찾기"**다. 좋은 문서를 찾아줘도, LLM이 그 문서를 *읽고 정확히 답하는지*(답변 충실도, answer-faithfulness)는 **별개**다(3단계). 진짜 "의미를 읽고 답한다"는 거기서 판가름 나며, 그건 **아직 측정 안 됐다.** → "의미 못 읽는 문제 없다"는 **아직 못 한다.** *찾기는 크게 해결, 읽고·답하기는 다음 숙제.*

## 8.6 웹 검색 적용성 (지식 vault ↔ 인터넷 web)
오늘 결과는 **주로 지식(vault RAG: 큐레이션 한국어 코퍼스)** 대상. 하지만 핵심 패턴(질문→의미→정본 수렴→근거)은 **웹에도 전이**된다 (Leo 통찰: "여러 사이트를 읽어 비슷하게 처리").

**전이되는 것 (vault=web 공통):**
- **Query Massage** — 웹에도 큰 이득 (원문보다 정본 검색어가 검색 API 결과↑, 외국어 질문→영/한 키워드도 동일).
- **조건부 리랭커** — 가져온 웹 스니펫/문단 재정렬에도 동일 정책.
- **Grounding Gate** — 웹은 **오정보가 많아 더 중요** (근거 약하면 "모름").

**웹 고유 (추가 레이어 — 별도 처리):**
| 축 | vault(지식) | web(인터넷) |
|---|---|---|
| 코퍼스 | 닫힘·큐레이션·**사전 임베딩** | 열림·무한·**질의시 실시간 fetch**(사전 색인 불가) |
| 신뢰 | 정본(통제) | **출처 품질 편차 큼**(오정보) → 출처 신뢰도·교차확인 |
| 최신성 | 고정 | **실시간성**(시세·뉴스·일정) → freshness 가중 |
→ "읽고 의미 처리"는 같지만, 웹은 **①실시간 fetch ②출처 신뢰 ③최신성**이 더 붙는다. **SIASIU의 핵심은 큐레이션 vault**(제품·성분·시술)이고, 웹은 보조(트렌드·시술후기 등)이며 *고신뢰 도메인이라 더 보수적으로* 쓴다.

## 8.7 ★ judge-통합 evidence-acquisition 층 (Leo+GPT 2026-06-24) — "검색 엔진"이 아니라 *판단 보강 시스템*
**핵심:** Search의 부족함은 "검색창이 없다"가 아니라 — *judge가 언제·무엇을·어디서·어떤 기준으로 검색하고, 결과를 어떻게 *안전하게* 판단에 반영하느냐*의 루프가 약하다.
★**별도 SEARCH 시스템 ❌ — 이 루프 = 자가성장 지식 루프**(`[[자가성장지식설계서]]` §7). 출처등급=*인용표준 화이트리스트*, candidate→reviewed→stable=*자가성장 staging*, 충돌 덮어쓰기 금지=*불변식 8* — *이미 있는 것 재사용.* **새로 짓는 건 ①트리거 ②쿼리플래너.**

```
judge가 모름/불확실 감지(아래 신호)
  → search_task{type, entity, persona_ctx, product_ctx, allowed/forbidden_sources, expected_output, memory_status_after}
  → query planner(canonical+alias+계열+persona × ko/zh/en)
  → retrieval(★fast: 내부 KB·수초 / slow: 웹·논문·비동기)
  → evidence grading(인용표준 화이트리스트: 규제/논문 S~A · 브랜드 B(=claim) · 쇼핑몰 C(검증대상) · 블로그 D · 출처불명 reject)
  → conflict check(기존 stable과·★안전주제 충돌→safety-first·confidence↓·덮어쓰기 ❌)
  → candidate memory(자가성장 staging·★바로 stable ❌)
  → judge 보수 주입(stable 강 · candidate 약 · marketing=검증대상 · 수치화 ❌)
  → 답변 → promotion queue(Opus 검증 후 승격)
```

**① search_needed 트리거 (judge의 "모른다" 신호 — ★RV-7 수정으로 오늘 *입력이 생김*):**
미매핑 높음(`insufficient_mapping`·`confidence: low`) · 새 ester/derivative · atom 있으나 임신/수유 정보 없음 · 브랜드 claim↔성분 충돌 · candidate edge 근거 부족 · 공개농도 없는데 강한 효능 claim · 리뉴얼/버전 불확실. → *stable·충분이면 search_needed: false*(불필요 검색 ❌).

**② query planner:** 일반 AI "retinyl propionate breastfeeding safe" ❌ → *구조적·다국어:* `retinyl propionate topical lactation safety` · `retinyl esters pregnancy dermatology review` · `视黄醇丙酸酯 哺乳期 外用 安全性` · `레티닐프로피오네이트 수유 화장품 안전성`. (canonical+alias+계열+persona+언어 조합.)

**judge 주입 불변식(= Judge-Locked·grounding gate 재사용):** stable evidence만 강하게 / candidate는 confidence 낮게 / marketing claim은 *검증 대상으로만* / 충돌 시 *safety-first·단정 금지* / 검색 결과 *바로 사실 확정 ❌·바로 stable ❌*.

**★구현 순서 (검색 *실행* 전에 "언제·무엇을"부터):**
`test_search_triggers.py`(검색 필요 감지·구조화) → query planner → evidence grading → conflict check → candidate memory → *그다음* 실제 retrieval 연결. *외부 검색을 먼저 짓지 않는다.*
