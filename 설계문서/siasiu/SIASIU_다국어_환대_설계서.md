## 0. 한 줄 / 원칙
**시스템 언어(설정)로 먼저 환대하고, 고객이 다른 언어로 물으면 그 언어로 답한다 — 살짝 섞인 건 무시.** 출력은 *네이티브 생성*(번역체 ❌, 톤이 심장). 느려도 정확하게.

## 1. ★ 두 언어 신호 (구분이 전부)
| | **시스템 언어** (설정) | **발화 언어** (메시지) |
|---|---|---|
| 무엇 | 고객이 *설정한* 언어(지역·선호) | 그 메시지에서 *실제로 쓴* 언어 |
| 신뢰도 | **확실**(명시 설정) | 추정(특히 라틴문자 모호) |
| 구동 | **첫 인사·UI·설문·설정·채팅 기본** | *명확히* 다른 언어로 물으면 그 답만 미러 |
| 규칙 | 베트남 설정 → 베트남어로 *먼저* 환대 | *살짝 섞인* 외국어(단어 몇 개)는 무시 — 주 언어 유지 |

**왜 시스템 언어가 1차인가:** ★베트남어·말레이어·영어는 *모두 라틴문자*라 발화만으론 구분 못 함. *설정 언어가 유일하게 확실한 신호.* 발화 감지는 *보조*(명확한 스크립트 전환일 때만 override).

## 2. 시스템 언어 — 소스·저장·전달
- **★소스(Leo 확정):** 첫 진입 시 **언어 선택 단계(필수)** — 단 *브라우저/시스템 locale를 감지해 기본값을 미리 선택*해두고, 고객이 확인하거나 바꾼다. → *편의 제공 + 우리도 확실한 언어정보 확보*(항상 명시적). ("태국 고객에게 영어 화면이 나오면 당황한다" — 스마트 기본값으로 방지, 단 최종은 고객이 명시.)
- **저장:** 프런트 `store.lang`(영속) + 로그인 시 서버/brain에 전달(`system_lang`). 회원이면 프로필에 영속(재방문 일관).
- **전달:** `brain.chat(user_id, message, name, system_lang=...)` — 인사·채팅 기본 언어로 사용. (현재 미배선 → 단계1.)
- **지원 언어(MVP 7):** `ko · en · zh · ja · th · vi · ms` (한·영·중·일 + 태국·베트남·말레이 — SEA 우선).

## 3. 언어 정책 (무엇을 어느 언어로)
- **인사·UI·설문·설정 = 항상 시스템 언어.** (베트남 설정 → 전부 베트남어. 한국어 절대 X.)
- **채팅 답변 언어 = 결정 규칙(우선순위):**
  1. 발화가 *시스템 언어와 다른 스크립트로 명확*하면(태국·한글·CJK·가나) → **그 발화 언어로** 미러.
  2. 발화가 *라틴문자*인데 시스템 언어도 라틴(en/vi/ms)이면 → 모호 → **LLM 언어판정**(아래 §4). 명확히 다른 언어면 미러, 아니면 **시스템 언어 유지**.
  3. *살짝 섞임*(주 언어 + 외국어 단어 몇 개) → **무시**(시스템 언어/주 언어 유지).
  4. 판정 실패·모호 → **시스템 언어**(안전 기본).

## 4. 발화 언어 감지 (무휴리스틱 — 스크립트 결정론 + LLM 보조)
- **스크립트 결정론**(`_detect_lang`): 한글·CJK한자·가나·태국문자(U+0E00–0E7F)는 *문자로 확실히* 구분 → 즉시 판정.
- **라틴 모호(en/vi/ms 등)**: 문자비율론 구분 불가 → **LLM 언어판정 1콜**(`{"lang":"en|vi|ms|..."}`, 무휴리스틱 의미판단). *시스템 언어와 같으면 콜 생략*(기본 유지 — 비용↓).
- **혼합 임계(살짝 섞임 무시):** 발화의 *주 스크립트/언어가 지배적*(예: 비라틴 문자 비율 충분, 또는 LLM이 "주 언어 X, 일부 외국어 단어"로 판정)이면 주 언어 유지. 단어 몇 개의 외국어는 전환 트리거 아님.

## 5. 인사 (첫인상의 심장)
- **시스템 언어로 *네이티브 생성*** — `_soft_refuse`처럼 LLM이 *그 언어로 따뜻하게·varied* 생성(번역체 ❌). 7개 언어 다 자연스러운 환영.
- **이름 호명**(있으면) + 재방문이면 지난 주제 회상(메모리) — 단 *시스템 언어로.*
- 폴백: LLM 불가 시 언어별 최소 템플릿(짧은 환영).

## 6. ★ 근본 원리 (왜 어렵고, 어떻게 푸나)
- **프롬프트 언어 지배:** LLM 출력 언어는 *지시*보다 *프롬프트의 지배 언어*에 끌린다. 우리 프롬프트는 ~한국어(PERSONA·KB근거·예시) → 외국어 질의에도 한국어로 샘(루프3에서 CLIR-EN on-domain 실측).
- **설계 긴장:** 내부 정본 = 한국어(Meaning Bridge — 검색·메모리 일관성, *좋은* 설계) ≠ 출력 언어(고객). 생성 컨텍스트(한국어)와 출력(고객어)이 다름.
- **★해법 = 프롬프트 현지화 (Leo 2026-06-23 결정: B 근본).** 루프 매트릭스(7×4=28, 23 PASS 82%)가 *워크어라운드의 fragile함*을 확정: ja·th·vi·ms 4/4 완벽이지만 **zh 3/4 FAIL**(system_lang=zh인데 메시지가 비-중국어면 한국어 누출·flaky) + en 섞임무시 1건. 근본 = 한국어 프롬프트 지배 → zh _LANGDIR 신호 불안정.
  - **현지화 구현:** 정본 PERSONA는 *한국어 1개*(소스). 비한국어 system_lang이면 **LLM이 그 언어로 번역해 캐시**(언어당 1회·메시지마다 비용 X). 정적 persona·말하는방식·grounding 래퍼 = 타깃 언어 / 동적(메모리·회피·KB근거) = 원문 유지 + "답은 고객 언어로". *프롬프트 대다수가 타깃 언어 → 모델이 자연히 그 언어로(zh도 안정).*
  - **+ 섞임 임계:** `_output_lang`이 *지배 비율* 충분할 때만 전환(한국어 단어 몇 개에 안 끌림).
  - *지양:* post-translate(한국어 생성→번역) = 언어 100%지만 *번역체로 톤 죽음* — 브랜드 심장에 반함. / 워크어라운드(네이티브 지시) = ja/th/vi/ms엔 충분하나 zh fragile → 현지화로 대체.

## 7. i18n 범위 (UI·설문·설정)
- 현재 `store.js` DICT = ko/en/zh만 → **th·vi·ms·ja 추가.**
- 대상: 온보딩·로그인·탭·MY드로어·설정·**만족도 설문**·고정 UI 문구. (플랜·법적 문구는 별도.)
- 동적(상담 답변)은 §3 정책으로 런타임 결정. 정적 UI는 DICT.

## 8. ★ 테스트 매트릭스 (루프 — 7언어 × 4과정)
각 언어(ko·en·zh·ja·th·vi·ms)마다:
1. **로그인 첫 인사** → 그 언어로 따뜻하게? (한국어 누출 0)
2. **설문/채팅 기본** → 그 언어로?
3. **중간 전환** → 다른 언어(명확)로 물으면 그 답만 그 언어로?
4. **살짝 섞임** → 주 언어 + 외국어 단어 몇 개 → 주 언어 유지(전환 안 함)?
→ LLM 판정 → 실패 수정 → 재시뮬. *green까지.* (정확도 우선 — 느려도 됨.)

## 9. 빌드 단계 (각 단계 후 루프테스트)
1. **시스템언어 배선** — store(자동감지+수동) → server → `brain.chat(system_lang)` → 저장. 채팅 기본 언어 적용.
2. **인사 네이티브 생성** — 시스템 언어로 따뜻하게(7언어).
3. **채팅 언어 정책** — §3 결정규칙 + §4 감지(스크립트+라틴 LLM+혼합무시). on-domain 한국어맥락 끌림 근본 수정(§6 현지화).
4. **i18n 확장** — UI·설문·설정 7언어(DICT).
5. **루프 매트릭스** — §8 7×4 → green.
