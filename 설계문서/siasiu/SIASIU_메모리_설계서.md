## 0. 한 줄
**대화 → 의미있게 정리(승격) → 재방문 시 정확히 회상.** 메모리가 곧 차별점. 정확도 우선 — *틀린 기억보다 "모름"이 안전.*

## 1. 봄 참고 (요지)
단기(대화맥락·스냅샷, 휘발) / 장기(에피소드=전 보존, 승격사실=영속). 단기→장기 승격 시 휘발성·잡담은 SKIP. **봄은 파일/마크다운(로컬 단일기기), 우리는 DB(웹·멀티유저).** 구조는 같고 매체·견고성이 다름.

## 2. 메모리 층 (SIASIU)
**단기 (세션 작업기억, 휘발):** 대화 맥락("그거·아까") + 세션 스냅샷(이번 상담 결과).
- ★ **세션 연속성:** 단기는 내부적으론 휘발이지만, **재진입 시 에피소드(장기)에서 최근 대화를 복원**해 채팅을 이어붙인다 → 사용자는 끊김을 못 느낀다. (페이지 이동·앱 계속 켜둠 모두 대응.)

**장기 (영속, DB):**
- **에피소드:** 모든 대화 턴을 **정확한 타임스탬프 + 순서**로 보존(삭제 금지). 늘 읽진 않지만 원본·복원·재추출용.
- **★ 평생 사실(memory_fact):** 그 사람의 영속 정보. **AI가 상담 때 늘 먼저 보는 것.**

## 3. ★ 사실 타입 레지스트리 ("모델", 전역)
사람마다 *항목 개수는 다르다*(듬성듬성 — 해당되는 것만). 그러나 **타입(이름·종류)은 전역 공유 카탈로그**로 통제한다 → 모두에게 *같은 이름·같은 방식*으로 추가.
- 예 타입: `avoid_ingredient · allergy · skin_type · personal_color · goal · age_range · budget · lifestyle · pregnancy_nursing · reaction · concern · preference`.
- **추출 규칙:** LLM은 **먼저 기존 canonical 타입에 매핑**(이름 일관). 진짜 새로운 것만 *새 타입 제안*.
- **★ 처음부터 통제 (Leo 2026-06-20 — 스키마는 first-class, 나중에 못 고치고 잊어버림):** canonical 타입 + 정규화/별칭 매핑을 **day 1부터** 내장. 새 타입 자동 채택 ❌ → 먼저 정규화·중복제거(`avoid_ingredient` = `피하고싶은성분` = `ingredient_to_avoid` → 하나의 canonical). **"같은 뜻 다른 타입"이 둘 다 공식이 되는 일 자체를 막는다.**
- 검토 게이트: **MVP = 우리(Leo+Claude) 가볍게 확인 / 프로덕션 = 같은 게이트 스케일.** 통제 메커니즘은 처음부터 동일(느슨하게 시작 안 함).

## 4. 평생 사실 — 형식 (하이브리드)
**구조화 항목(typed) + 짧은 문장 노트.** 둘 다.
- **구조화 항목** — 안전·핵심·조회용. 예: `avoid_ingredient="고농도 레티놀"`, `skin_type="건성·민감"`, `personal_color="여름 쿨"`.
- **문장 노트** — 항목에 안 담기는 뉘앙스·패턴. 예: "수면 무너진 주에 피부 예민", "무향·젤 선호".
- 각 사실 필드: `type · value · note · confidence · as_of(날짜) · status(active/superseded) · source_episode_id`.

## 5. 승격 (추출) — DeepSeek + corroboration
대화에서 DeepSeek이 **영속 사실만 구조화 추출** → memory_fact upsert. 게이트: 휘발성·잡담 SKIP(장기 오염 방지).
- **★ 2번 규칙 (봄에서 가져옴):** 단발 언급은 *확정 사실로 굳히지 않음* — 낮은 confidence(hypothesis). **같은 사실 2번 보면 confidence↑(confirmed).** (잡담 한 번을 사실로 박는 사고 방지.)
- 정확도 우선: confidence 낮으면 회상에 약하게, 단정 금지.

## 6. 시간 · 최신화 · 충돌 (★핵심)
- **모든 사실·노트에 `as_of` 날짜.** "6개월 전 구매"는 *오래됨*으로 인식 — 실효 지난 걸 안 우김.
- **상태 변화 = 갱신:** "여드름→회복" 같은 문장은 **구조항목(skin_type)을 갱신하는 신호.** 옛것 *superseded*(날짜째 보존) + 새 사실. *여드름을 영원히 안 박음.*
- **★ 망각/만료(decay, 봄에서 가져옴):** 오래 재확인 안 된 사실은 자동 강등(confidence↓·stale 표시). `as_of`와 결합.
- **★ 충돌 우선순위 (항목 vs 문장 / 옛 vs 새):**
  1. **최신 날짜 우선** (먼저 적용 — 모순 방지).
  2. **구조항목 = 정본(authority)** for 하드 사실. 문장은 *항목을 갱신하는 입력*이지 권위가 아님.
  3. **안전 비대칭:** 회피·알러지 *추가*는 쉽게, *제거*는 **명시 확인 후에만.** (애매한 문장으로 알러지 안 지움.)
  4. 그래도 애매하면 **단정 말고 되묻거나 "모름".**

## 7. 회상 + 재진입 인사 + 추천 칩
**회상:** 새 세션/재진입 시 **memory_fact(active) 주입** → "이 고객: 레티놀 회피(⚠), 무향 젤 선호, 여름쿨, 목표 장벽회복…" → DeepSeek이 그 위에서 답.

**★ 재진입 인사 (창이 비면 *무조건* — Leo):** 화면이 비어 재진입하면 반드시 **따뜻하게** 인사해 안심시킨다(안 그러면 "날 잊었나" → 신뢰 붕괴).
- **톤 = K-드라마급 환대 + 공감** (고민이 있어 오는 사람). 따뜻·친절·이름 호명. ("이어가요?" 같은 사무적 표현 지양.)
- **"다 기억해요" 금지 (행동진실):** *구체적으로 진짜 기억하는 것만.* 사실 없으면 "처음 뵙는 것 같아요"(헛것 금지).
- **다양하게:** 하드코딩 템플릿(한 언어 작성 → 사용자 언어 번역) + 실제 회상 사실 1개 채워 랜덤.
- 템플릿(예):
  - *재방문:* "{이름}님, 다시 오셨네요 😊 {기억}, 그동안 마음 많이 쓰이셨죠. 오늘은 좀 어떠셨어요?" / "{이름}님 기다리고 있었어요. {기억} 그 뒤로 좀 괜찮아지셨어요? 천천히 같이 봐드릴게요."
  - *신규:* "{이름}님, 만나서 정말 반가워요 😊 오늘 상담 도와드리게 되어 기뻐요. 피부든 건강이든 편하게 뭐든 말씀해 주세요 — 친절히 도와드릴게요."

**★ 추천 질문 칩 (빈 화면마다 *무조건* — ChatGPT/Claude식):** 입력창 *위*에 칩 노출 → 사람들이 *뭐가 되는지*(특히 제품검색) 알게. 화면 클리어 시 항상.
- **순서(고정):** ① 무조건 첫째 = **제품 검색** ② 둘째 = **이 사람 관련 문제(기억 기반 개인화)** ③ 셋째 = 또 하나 관련. **총 3개, 넘치면 가로 슬라이드.**
- 각 칩 = **굵은 제목 + 회색 부제**(GPT 형식). 누르면 그 질문이 입력돼 상담 시작.
- 예(재방문·레티놀자극): `[제품 찾기 / 내 피부에 맞는 제품]` · `[지난번 그 고민 / 레티놀 자극, 대안 찾기]` · `[내 피부 진단 / 요즘 왜 이런지]`
- 예(신규·여드름): `[제품 찾기 / 나에게 맞는 제품]` · `[여드름 상담 / 근본 원인부터]` · `[퍼스널컬러 / 나에게 맞는 색]`

## 7b. ★ 체크인 ↔ 메모리 흐름 (Leo 2026-06-20)
홈 체크인 카드 = **메모리가 스스로 최신을 유지하고, 고객을 다시 오게 만드는 엔진.**

**층 구분 (확정):**
- **단기** = *채팅 세션에서만* 쓰는 라이브 작업버퍼(휘발). 그 외 용도 없음.
- **에피소드** = 모든 턴 raw·영구. **장기 사실** = 승격된 영속 정보.

**체크인 답은 어디로? (예: "좋아졌어요")**
- → **에피소드**(시스템 질문 + 고객 답 = 한 턴, raw·영구) **+ 장기 사실 갱신**(상태변화: 고민 추세=개선 / 반응=잘맞음, `as_of` 갱신·옛것 supersede).
- **단기 아님** — 세션 끝나도 남아야 하니까.
- **★ 구조화 답(버튼)이라 LLM 추출 불필요 → 코드로 바로 매핑**(좋아졌/그대로/더안좋아 → 결정론적 갱신). *자유 채팅만 DeepSeek 추출.* (싸고·정확·확실.)

**체크인 질문 생성 = 3갈래로 섞음:**
- **① 최근 대화 주제**(recency, 1~2): "지난번 '○○' 얘기, 그 뒤로 어때요?"
- **② 장기 활성 고민**(추적 중 미해결, 1~2)
- **③ 오래된 사실 재확인**(★decay refresh, 1): `as_of` 오래된 사실 → "예전에 ○○라 하셨는데 요즘도요?" → 답하면 갱신/supersede.
- **우선순위:** 안전(회피·알러지 변화) > 미해결 고민 > 최근 주제 > 오래된 refresh.

**★ ③번 = §6 decay의 실행 메커니즘:** 사실이 조용히 썩는 게 아니라 **체크인이 끄집어내 재확인**한다 = "오래된 사실을 *어떻게* 다시 확인하나"의 답.

**닫힌 루프:** 메모리 → 질문 생성(체크인) → 고객 답 → 메모리 갱신 → 다음 체크인. = **자가정비 + 리텐션 엔진.**

## 8. 안전 (코드로 강제 — 봄보다 단단)
- **안전 핵심(알러지·회피성분)은 구조항목 + 코드 하드체크.** LLM 기억에만 안 맡김 — 추천에 회피성분 들어가면 **코드가 차단.** (건강·회사 제품이라 "잊으면 사고".)
- 출처 보존(source_episode_id) → 검증·정정 가능. 가격·휘발성은 사실에 안 박음.

## 9. 데이터 모델 (MVP — 단일 SQLite DB)
```sql
user(user_id, name, lang, created_at)
episode(id, user_id, ts, role,            -- 모든 턴, 영속, 순서 보존
        text)
memory_fact(id, user_id, type, value, note,
            confidence, as_of, status,     -- status: active|superseded
            source_episode_id, created_at, updated_at)
fact_type_registry(type, label, is_safety, -- 전역 "모델"(타입 카탈로그)
                   norm_aliases, status)   -- status: active|proposed(검토대기)
session_snapshot(user_id, session_id, json)-- 세션 동안만(단기)
```
프로덕션: MySQL, 같은 구조.

## 10. MVP 흐름 (기억 루프)
```
[준비]  온보딩 입력 + 가짜 시드 → memory_fact 초기화 + 소형 KB
[대화]  질문 → active 사실 + KB 주입 → DeepSeek 답(페르소나·grounding·안전 하드체크)
[정리]  세션 끝 → DeepSeek 추출(2번 규칙) → memory_fact upsert(supersede·as_of) → episode 저장
[재진입] 창 비면 → 구체 회상 인사(다양·번역) → 최근 에피소드 복원해 대화 이어감
```
**완료 기준:** 같은 (가짜)유저 1차 상담 → 종료/재진입 → *지난 내용을 정확히·구체적으로 회상*. 근거 없으면 "모름". 회피성분은 코드가 차단.

## 11. ★ 메모리 도메인 — 일반상담 ≠ 고객상담 (쓰기 격리 · 읽기 공유) (Leo 2026-06-21)
고객상담(CS)이 추가되며 메모리를 **2 도메인**으로 나눈다 — *오염 방지 + 완전 파악*을 동시에.

| | **뷰티/건강 메모리** | **구매/주문 메모리** |
|---|---|---|
| 쓰는 주체 | 일반상담(시아) | **고객상담(CS)** |
| 내용 | 피부·고민·회피·반응·목표·퍼스널컬러·체크인 | 주문·배송·환불/교환·컴플레인·사용중 제품·재구매 주기·문의이력 |
| 읽는 주체 | **둘 다** | **둘 다** |

- **★ 쓰기 격리(write-isolation):** 각 채팅은 *자기 도메인 사실만* 쓴다. advisory 추출 → 뷰티 facts · CS 추출 → 구매 facts. → CS 잡담이 건강 프로필 안 더럽히고, 상담 잡담이 주문기록 안 어지럽힌다. (가드레일 §11 메모리 격리의 구체화.)
- **★ 읽기 공유(read-sharing):** 두 시스템 모두 *양쪽 도메인을 다 읽는다.* CS도 "민감성"을 알고, 시아도 "지난주 산 레티놀"을 안다 → **한 사람을 완전히** 해결.
- **구현(스키마):** `memory_fact`·`episode`에 **`domain` 필드(advisory | cs)** 추가.
  - 추출 라우팅: advisory 세션 → `domain=advisory`만 기록 / CS 세션 → `domain=cs`만 기록.
  - 컨텍스트(읽기): 프롬프트 구성 시 **둘 다** 로드(관련도 필터).
  - 재진입 replay: 도메인 스코프(상담챗=advisory 에피소드, CS챗=cs 에피소드). 단 *교차 읽기*는 허용.
- **새 fact 타입(구매 도메인 — 레지스트리 확장):** `order` · `delivery`(배송상태) · `refund_exchange` · `complaint` · `product_in_use`(사용중 제품) · `repurchase_cycle`(재구매 주기).
- **단일 신원 위:** 두 도메인 모두 *같은 user_id*에 매달림 → 한 사람 = 통합 인격(읽기)·둘로 분리(쓰기). (신원 = Phase 2.)
- **★ 도메인 핸드오프 (Leo 2026-06-21) — 오염 방지의 실행 장치:** 한 채팅에서 *다른 도메인 주제*가 나오면 **넘긴다.** 예: 시아(일반상담)에게 *구매 피드백·환불·배송* 얘기를 하면 → **고객상담(CS)으로 부드럽게 링크 핸드오프** → 그 대화는 *구매 메모리(domain=cs)* 에 기록(뷰티 메모리 오염 ❌). 반대도 동일. 즉 *라우팅이 쓰기 격리를 실제로 지켜주는 장치.* (핸드오프도 따뜻하게 — "그건 제가 주문 담당이랑 바로 연결해드릴게요 🙂")
- **스케줄:** 구현은 **Phase 7B(고객상담)** 와 함께. 단 `domain` 필드는 *지금 넣어두면 forward-compatible*(기본 advisory).
